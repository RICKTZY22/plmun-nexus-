◆ DFD Level 2

Process 3 — Request Management (Detailed)

Decomposition of P3 into 5 sub-processes showing the complete borrow-approve-return lifecycle.

#Sub-ProcessInputProcessing LogicOutput
P3.1 Validate & Submit Request Item ID, Quantity, Purpose, Priority from User Check stock ≥ requested qty; check user role ≥ item access_level; check user not flagged; snapshot item_name New Request record (status=PENDING) → D3
P3.2 Review & Decide Pending request from D3 + Staff/Admin decision Block self-approval (approved_by ≠ requested_by); set approved_at timestamp; calculate expected_return from item.borrow_duration Status → APPROVED or REJECTED (with reason) → D3
P3.3 Deduct Inventory Approved request with quantity F('quantity') - approved_qty (atomic, race-safe); auto-set item status if qty hits 0 Updated stock in D2; audit log entry → D4
P3.4 Process Return Return action from Staff + returned_at timestamp F('quantity') + returned_qty (restock); set status=RETURNED; check if return is overdue → increment user overdue_count & flag if ≥3 Restocked item in D2; updated request → D3; overdue notification → D5
P3.5 Broadcast Notifications Status change event + recipient calculation bulk_create notifications for: requester + all Staff/Admin; set type based on action (STATUS_CHANGE, OVERDUE, COMMENT) Notification records → D5
