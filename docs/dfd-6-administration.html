User & System Administration

Admin-only operations: user management (role assignment, activation/deactivation), profile management, system audit logs, and maintenance tasks like clearing request history.

Sub-Process 6.1
Manage Users (Admin)
Admin can view all registered users, change their roles, and view their activity statistics. This is the only way to promote a Student to Staff or Admin.

Sub-Process 6.2
Activate / Deactivate User
Admin can deactivate a user account (prevents login) without deleting it. Useful for graduated students or suspended accounts.
🔑 Admin
Toggle active action
6.2

D1: Users
Deactivate Effect: User's is_active=False → Process 1.1 (Validate Credentials) rejects login with "Account deactivated"
Reactivate: Admin can re-enable the account anytime. All previous data (requests, history) is preserved.
vs. Delete: Deactivation is reversible. Deletion is permanent and cascades to remove all user data.

Sub-Process 6.6
System Maintenance
Staff/Admin can clear completed/returned/cancelled requests to keep the active view clean. Uses soft-delete (is_cleared=True) — data preserved for audit.
🛡️ Staff / Admin
Clear history action
6.6
Clear
History
is_cleared = true
D3: Requests
6.6
Clear
Audit: "History Cleared"
D6: Audit Logs
Scope: Only affects requests WHERE status IN ('RETURNED', 'REJECTED', 'CANCELLED'). Active/pending requests are never affected.
Soft Delete: Sets is_cleared=True. These requests are filtered out of the normal view but remain in the database for reports and audit.
vs. Hard Delete: No data is permanently removed. Admin can verify historical data through audit logs and reports.
